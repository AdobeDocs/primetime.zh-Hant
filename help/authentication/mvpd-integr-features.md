# MVPD整合功能

>[!NOTE]
>
>此頁面上的內容僅供參考之用。 使用此API需要來自Adobe的目前授權。 不允許未經授權的使用。

## 概觀 {#mvpd-int-features-overview}

Adobe Primetime驗證支援各處電視的新興標準。 它符合 **CableLabs OLCA （線上內容存取）規格**，提供從線上來源傳送視訊給付費電視客戶的技術需求和架構。 Adobe於2011年6月參與了CableLabs的聯合Interopt測試專案，並通過了服務供應商實作的測試程式。  Adobe也是 **OATC （開放式驗證技術聯盟）** 並參與幾個小組委員會的規格起草專案，作為該機構的一部分。

在說明Primetime驗證OLCA合規性以及Adobe參與OATC後，也務必注意Primetime驗證實際上是「不受通訊協定限制」的。  但在TVE時代的這個階段，Primetime驗證絕對是以OLCA標準為導向。  這些標準說明不同TVE播放器（程式設計師、MVPD和服務提供者）實作TVE功能的目前議定方式。 下表列出許多這類功能，並提供相關頁面的連結，其中提供詳細資訊和如何實作這些功能的範例。

表格中的資訊旨在推動Primetime驗證整合程式，使其在所有MVPD整合中功能一致。 優先順序欄將功能分為A、B和C：

* 答 — 這些是「必須具備」的功能，才能初次推出整合。
* B — 這些是初始整合的重要增強功能，將在初始推出後新增。
* C — 這些是整合的進一步增強功能，可在「B」要求後實作。

| 1.3 | 使用者ID持續性 | MVPD會提供經過模糊處理的永久UserID。 | A |  |

| 1.5 | 裝置最佳化的登入使用者體驗(UX) | MVPD支援調整登入畫面大小，以符合檢視所使用的裝置尺寸。 | A |  |

| 1.7 | [服務提供者範圍](/help/authentication/serv-provider-scoping.md) | MVPD支援在AuthN要求中傳遞品牌識別碼（要求者值）。 | A- | 如此可讓服務提供者特定的登入體驗。 |
| 1.8 | [多頻道授權](/help/authentication/mvpd-preflight-authz.md#preflight-multich-authz) | MVPD支援程式設計師在單一授權要求中傳送多個通道。 | B+ |  |
| 1.9 | iFrame或「JS快顯」式驗證 | 程式設計師能夠將登入流程整合到iFrame或快顯視窗體驗中，而不是HTTP重新導向。 | B |  |
| 1.10 | [程式設計師啟動的登出](/help/authentication/usecase-mvpd-logout.md) | 檢視器在程式設計師的網站或應用程式上以及在MVPD上有已驗證的工作階段。 檢視器可以從程式設計人員網站啟動同盟登出，登出也會清除MVPD入口網站上的工作階段。 | B | 從程式設計人員的角度來看，確保共用電腦受到更嚴密的保護，避免誤用。 |
| 1.11 | [自訂授權錯誤訊息](/help/authentication/error-reporting.md) | MVPD會傳遞自己的自訂錯誤字串，該字串適合程式設計人員的同盟網站或應用程式顯示給使用者。 | B | 啟用向上銷售案例 |
| 1.12 | **驗證回應的使用者中繼資料** | MVPD AuthN回應可包含使用者中繼資料，作為軟體權利檔案流程期間使用者體驗個人化的提示。 此要求可啟用從MVPD到程式設計人員的家長控制提示。 | B- |  |
| 1.13 | MVPD啟動的驗證 | 檢視器在MVPD入口網站上完成成功的AuthN工作階段，然後導覽至程式設計師的TVE網站。 系統不會提示使用者輸入MVPD選擇器，且會自動驗證使用者。 | B- |  |
| 1.14 | 使用者ID範圍 | MVPD UserID應該以兩種形式出現 — 一個以程式設計人員為範圍，另一個以整個Adobe為範圍，以防欺詐。  這可讓Adobe在不進一步加密/模糊化的情況下共用程式設計人員範圍的MVPD UserID。 | C |  |
| 1.15 | 資產授權 | AuthN完成後，AuthZ就可以在背景執行，方法是傳遞結構化資料，包括網路、節目、資產、家長監護分級等。 如此一來，程式設計師對MVPD的每個AuthZ呼叫都會啟用家長監護。 | C |  |
| 1.16 | MVPD啟動的登出 | 檢視器在程式設計師的網站或應用程式上和MVPD都有已驗證的工作階段。 檢視器可以從MVPD的網站啟動同盟登出，也會清除所有同盟程式設計師網站上的工作階段。 | C |  |
| 1.17 | 授權義務 | MVPD在AuthZ回應中提供其他條件，例如記錄或更新的家長監護分級上限(OLCA)。 | C |  |
| 1.18 | IP位址內容 | MVPD需要明確傳遞IP位址。 若是呼叫位於伺服器端的AuthZ，這會提供MVPD詐騙追蹤資訊，讓您瞭解使用者來自AuthZ呼叫何處。 | C |  |
| 1.19 | Token持續存在存留時間(TTL)值會以動態方式定義 | MVPD可以透過回應中的屬性動態設定Primetime驗證Token的TTL，以便Adobe不在TTL變更的回圈中。 | C- |  |
| 1.20 | 裝置型別 | MVPD支援在AuthN或AuthZ要求中傳遞裝置型別。 此屬性會通知MVPD將會使用內容的裝置性質，好讓他們可以調整AuthN或AuthZ權杖的TTL，以符合他們自己對裝置的安全性考量。 | C- | 這有助於無使用者端平台，在其中可能很難在Primetime驗證端將每種裝置型別公開為設定設定。 |
