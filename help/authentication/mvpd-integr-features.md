# MVPD整合功能

>[!NOTE]
>
>此頁面上的內容僅供參考。 使用此API需要來自Adobe的當前許可證。 不允許未經授權使用。

## 概述 {#mvpd-int-features-overview}

Adobe Primetime認證支援新興的TV Everywhere標準。 它符合 **CableLabs OLCA（線上內容訪問）規範**&#x200B;它提供了從線上源向付費電視客戶交付視頻的技術要求和體系結構。 Adobe於2011年6月參與了CableLabs的聯合互動測試項目，並通過了服務提供商實施的測試流程。  Adobe也是 **OATC（開放認證技術聯盟）** 作為該機構的一部分，參加了幾個小組委員會的規範起草項目。

在介紹了黃金時間驗證OLCA合規性以及Adobe參與OATC之後，還必須指出，黃金時間驗證實際上是「協定不可知」。  但是在TVE時代的這個階段，黃金時段認證的確是面向OLCA標準的。  該標準規定了目前為不同的TVE參與方（程式設計師、MVPD和服務提供商）實施TVE功能所商定的方法。 這些功能中的許多都列在下表中，其中包含指向相關頁面的連結，這些頁面提供了如何實施這些功能的詳細資訊和示例。

表中的資訊旨在推動黃金時段身份驗證整合過程在所有MVPD整合中實現一致的功能。 優先順序列將功能分為A 、 B和C :

* A — 這些是「必須具備」的功能，用於初始推出整合。
* B — 這些是對初始整合的重要增強，將在初始推出後添加。
* C — 這些是對整合的進一步增強，可在&quot;B&quot;要求之後實施。

| 1.8 | [多通道授權](/help/authentication/mvpd-preflight-authz.md#preflight-multich-authz) | MVPD支援程式設計師在單個授權請求中發送多個通道。 | B+ |  |
| 1.9 | 基於iFrame或「JS彈出」的身份驗證 | 程式設計師能夠將登錄流整合到iFrame或彈出式體驗中，而不是HTTP重定向。 | B |  |
| 1.10 | [程式設計師啟動的註銷](/help/authentication/usecase-mvpd-logout.md) | 查看器在程式設計師網站或應用程式上和使用MVPD都具有經過驗證的會話。 查看器可以從程式設計師站點啟動聯合註銷，註銷還會清除MVPD門戶上的會話。 | B | 確保從程式設計師的角度保護共用電腦免受誤用。 |

| 1.13 | MVPD啟動的身份驗證 | 查看器在MVPD門戶上完成成功的AuthN會話，然後導航到程式設計師的TVE站點。 系統不提示用戶輸入MVPD選取器，並會自動驗證用戶。 | B |  |

| 1.15 | 基於資產的授權 | 一旦AuthN完成，AuthZ就可以在後台進行，方法是傳遞可以包括網路、顯示、資產、家長控制分級等的結構化資料。 這可以對從程式設計師到MVPD的每個AuthZ調用啟用家長控制。 | C |  |
| 1.16 | MVPD啟動的註銷 | 查看器在程式設計師的站點或應用上以及使用MVPD都具有經過驗證的會話。 查看器可以從MVPD站點啟動聯合註銷，該註銷也會清除所有聯合程式設計師站點上的會話。 | C |  |
| 1.17 | 授權義務 | MVPD在AuthZ響應中提供附加條件，如日誌記錄或刷新的最大家長控制分級(OLCA)。 | C |  |
| 1.18 | IP地址上下文 | MVPD要求顯式傳遞IP地址。 對於AuthZ，其中調用是伺服器端，這將提供有關用戶在AuthZ調用中來自何處的MVPD欺詐跟蹤資訊。 | C |  |
| 1.19 | 動態定義令牌持久性生存時間(TTL)值 | MVPD能夠通過響應中的屬性動態地設定黃金時段驗證令牌的TTL，因此Adobe超出了TTL更改的循環。 | C- |  |
| 1.20 | 設備類型 | MVPD支援在AuthN或AuthZ請求中傳遞設備類型。 此屬性通知MVPD將使用內容的設備的性質，以便他們可以調整AuthN或AuthZ令牌的TTL，以符合他們自己對設備的安全考慮。 | C- | 這對無客戶端平台很有幫助，在該平台上，很難將每種設備類型作為黃金時段身份驗證端的配置設定公開。 |
