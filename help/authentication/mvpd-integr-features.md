# MVPD整合功能

>[!NOTE]
>
>此頁面的內容僅供參考。 若要使用此API，必須具備目前的Adobe授權。 不允許未經授權使用。

## 概述 {#mvpd-int-features-overview}

Adobe Primetime驗證支援新興的TV Everywhere標準。 符合 **CableLabs OLCA（線上內容訪問）規範**，提供從線上來源傳送影片給付費電視客戶的技術需求和架構。 Adobe於2011年6月參與CableLabs整合測試專案，並通過服務提供者實作的測試程式。  Adobe也是 **OATC（開放驗證技術協會）** 並作為該機構的一部分，參與了小組委員會的若干規範起草項目。

在介紹了Primetime驗證OLCA合規性以及Adobe參與OATC後，還必須注意到Primetime驗證實際上「不受協定限制」。  但是在TVE時代的這個階段，Primetime認證的確以OLCA標準為導向。  該標準勾勒了當前為不同TVE參與者（程式設計師、MVPD和服務提供商）實施TVE功能而商定的方式。 下表列出許多這些功能，並附上相關頁面的連結，提供實作功能的詳細資訊和範例。

表格中的資訊旨在促進Primetime驗證整合程式在所有MVPD整合中達到一致的功能。 「優先順序」欄將功能排名為A、B和C:

* A — 這些是「必須具備」的功能，可用於初始轉出整合。
* B — 這些是初始整合的重要增強功能，將在初始推廣後新增。
* C — 這些是整合的進一步增強功能，可在「B」要求後實作。

|------|----------------------------------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|----------|---------------------------------------------------------------------------------------------------------------------------------------------------------|
| 1.1 | [程式設計師啟動的登錄](/help/authentication/authn-oauth2-protocol.md) | 查看器選擇MVPD並啟動來自程式設計師品牌的網站或應用程式的驗證(AuthN)流。 | A+ |  |

| 1.5 | 裝置最佳化登入使用者體驗(UX) | MVPD支援重新調整登入畫面的大小，以符合檢視所使用裝置的尺寸。 | A |  |
| 1.6 | 預設MVPD選擇器徽標 | MVPD會提供適當維度（112x33像素）的預設標誌的URL。 | A | 標誌應由MVPD托管，且應由CDN快取。 |
| 1.7 | [服務提供商範圍](/help/authentication/serv-provider-scoping.md) | MVPD支援在AuthN請求中傳遞品牌識別碼（請求者值）。 | A- | 這會啟用服務提供者專屬的登入體驗。 |
| 1.8 | [多通道授權](/help/authentication/mvpd-preflight-authz.md#preflight-multich-authz) | MVPD支援程式設計師在單個授權請求中發送多個通道。 | B+ |  |
| 1.9 | iFrame或「JS快顯視窗」驗證 | 程式設計師能夠將登錄流程整合到iFrame或快顯體驗中，而不是HTTP重定向。 | B |  |
| 1.10 | [程式設計員啟動的註銷](/help/authentication/usecase-mvpd-logout.md) | 檢視器在程式設計人員網站或應用程式上和MVPD上都有已驗證的工作階段。 查看器可以從程式設計師的站點啟動聯合註銷，註銷還會清除MVPD門戶上的會話。 | B | 從程式設計人員的角度確保共用電腦得到更多保護，避免被濫用。 |
| 1.11 | [自定義授權錯誤消息](/help/authentication/error-reporting.md) | MVPD會傳遞其自定義錯誤字串，該字串適合程式設計師的聯合網站或應用程式向用戶顯示。 | B | 啟用向上銷售方案 |
| 1.12 | **驗證回應上的使用者中繼資料** | MVPD AuthN回應可包含使用者中繼資料，該中繼資料在權限流程期間作為使用者體驗個人化的提示。 這一要求允許從MVPD向程式設計師提供家長控制提示。 | B- |  |
| 1.13 | MVPD啟動的驗證 | 查看器在MVPD門戶上完成了成功的AuthN會話，然後導航到程式設計師的TVE站點。 系統不會提示使用者輸入MVPD選擇器，且會自動驗證。 | B- |  |
| 1.14 | UserID範圍 | MVPD UserID應以兩種形式出現：一種是限定為程式設計師，另一種是限定為範圍的Adobe，以防欺詐。  這可讓Adobe共用程式設計師範圍的MVPD UserID，而無需進一步加密/模糊化。 | C |  |
| 1.15 | 資產授權 | AuthN一旦完成，AuthZ便可在背景執行，方法是視需要傳遞結構化資料，包括網路、節目、資產、家長控制評等等。 這可以對從程式設計師到MVPD的每個AuthZ呼叫啟用家長控制。 | C |  |
| 1.16 | MVPD啟動的註銷 | 查看器在程式設計師的站點或應用程式上和MVPD上都具有已驗證的會話。 查看器可以從MVPD的站點啟動聯合註銷，該註銷還會清除所有聯合程式設計師站點上的會話。 | C |  |
| 1.17 | 授權義務 | MVPD在AuthZ回應中提供其他條件，例如記錄或重新整理的最高家長控制評等(OLCA)。 | C |  |
| 1.18 | IP位址內容 | MVPD需要明確傳遞IP位址。 若為AuthZ（呼叫為伺服器端），這可提供MVPD欺詐追蹤資訊，了解使用者在AuthZ呼叫中來自何處。 | C |  |
| 1.19 | 動態定義的代號持續存留時間(TTL)值 | MVPD能透過回應中的屬性以動態方式設定Primetime驗證Token的TTL，使得Adobe不會進入TTL變更的回圈。 | C- |  |
| 1.20 | 裝置類型 | MVPD支援將裝置類型傳遞至AuthN或AuthZ請求。 此屬性會通知MVPD將使用內容的裝置性質，讓MVPD可調整AuthN或AuthZ權杖的TTL，以符合其本身針對裝置的安全性考量。 | C- | 這對無用戶端平台很有幫助，因為在該平台中，很難在Primetime驗證端將每個裝置類型公開為組態設定。 |
