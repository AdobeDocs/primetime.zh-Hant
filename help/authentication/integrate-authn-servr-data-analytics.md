# 將Primetime驗證伺服器端資料整合至Adobe Analytics

>[!NOTE]
>
>此頁面的內容僅供參考。 若要使用此API，必須具備目前的Adobe授權。 不允許未經授權使用。

Adobe Primetime驗證的客戶想在Adobe Analytics控制面板中查看Primetime驗證(Adobe Pass)伺服器端資料，以便更輕鬆使用。

資料將用於追蹤重要的TVE量度，例如每個MVPD的驗證轉換率、以MVPD使用者ID為基礎的不重複使用者，以及更多。

如果已存在用戶端實作，則此實作並非用來取代用戶端實作，因為在沒有訪客ID的情況下，無法追蹤超過以下特定事件的使用者活動。 如果客戶在傳遞呼叫時提供訪客ID，我們可以解除鎖定另一種類型的Analytics整合（即時），這種整合可將所有傳遞事件與現有客戶資料一起加入，如需此新類型可能整合的詳細資訊，請前往：&quot;[在Primetime驗證中使用Experience CloudID](/help/authentication/exp-cloud-id-authn.md)&quot;

| 已請求AuthN | 啟動的驗證流數 |

| AuthN確定 | 用戶成功獲取的驗證令牌數 |
| 已請求AuthZ | 嘗試的授權數 |
| AuthZ確定 | 成功授權數 |
| AuthZ失敗 | MVPD在應用程式級別拒絕的授權數 |
| 播放要求 | 產生的短媒體代號數（與播放請求數同化） |
| 請求註銷 | 已啟動的註銷流數 |
| 註銷已完成 | 成功完成的註銷流數 |
| 註銷失敗 | 失敗註銷流數 |
| 已請求預授權 | 起始的預授權流數 |
| 預授權確定 | 使用預先授權的資源成功預授權事件的數量 |
| 拒絕預授權 | 具有被拒絕預授權的資源的預授權事件數 |
| 預授權失敗 | 失敗的預授權事件數 |

| Adobe Analytics名稱 | 說明 |
|------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 管道 | 用於執行權限要求的要求者ID |
| MVPD | 負責授與使用者權限的MVPD |
| 代理 | 代理MVPD（直接整合的「直接」） |
| SDK類型 | 使用的用戶端SDK(Flash、HTML5、Android原生、iOS、無用戶端等) |
| SDK版本 | Adobe Primetime驗證用戶端SDK的版本 |
| 資源ID | 授權請求中涉及的實際資源標題(從MRSS裝載中提取，作為項目/標題（如果提供）) |
| AuthZ錯誤類型 | 失敗的原因，如Adobe Primetime驗證報告 <br/> 以下是最常見的值 <br/> **noAuthZ** = MVPD回覆說，使用者的套件中沒有管道<br/> **網路** =我們無法連線MVPD（MVPD在呼叫當下發生問題，但未回覆）<br/> **norefreshtoken** =這嚴格適用於OAuth實作，如果使用者因故變更其密碼或MVPD拒絕密碼，可能會導致此錯誤。 這通常會產生新的驗證<br/> **不匹配** =如果要求來自與具有驗證Token的裝置不同的裝置。 如果使用者嘗試欺騙系統，但大部分都發生在舊版JavaScript SDK的內容中，而裝置ID在計算中使用的是IP位址。 如果用戶在家中觀看TVE，然後在工作時，將觸發此錯誤，他們將必須再次驗證<br/> **無效** =無效請求，缺少或無效參數<br/>  **authzNone** =程式設計師能夠拒絕特定通道xMVPD組合的授權。 這是由程式設計人員可存取的後端API所觸發<br/> **欺詐** =這是我們這邊的保護機制。 如果用戶失敗授權，然後在短時間（秒）內再次請求授權，我們將直接拒絕呼叫。 當程式設計師在實施中出現錯誤，如果失敗，就會不斷要求授權時，就會發生這種情況。 |
| 代號類型 | 當AuthZ全部和AuthN全部建立代號時，我們必須知道降級測量所造成的結果。<br/> 它們是：<br/> &quot;normal&quot; =正常情況<br/> &quot;authnall&quot; =啟用AuthN All時<br/> &quot;authzall&quot; =啟用AuthZ時<br/>  &quot;hba&quot; =啟用HBA時 |
| 無客戶端設備類型 | 裝置平台（替代），目前用於無用戶端。<br/> 值可以是：<br/> 不適用 — 事件並非源自於無用戶端SDK<br/> 未知 — 由於 **無用戶端API** 為選用，則有些呼叫不包含任何值。<br/> 透過 **無用戶端API**. 例如xbox、appletv和roku。 |
| MVPD使用者ID | 取代Cookie型訪客ID |


## 詳細資料 {#details-int-authn-analyt}

* 量度會透過Adobe Analytics Data Insertion API在特定報表套裝中插入，並以事件為依據
* 插入將被批次，每30分鐘傳送一次。 因此，報表必須加上時間戳記
* 每個客戶將有一或多個報表套裝。 一個要求者ID（管道）將僅對應至一個報表套裝。 多個請求者ID只能對應至一個報表套裝。
* 可提供歷史資料，但由於流量/效能考量，需要特別謹慎處理。
* 不重複訪客變數會設為MVPD使用者ID
* 無法設定事件和eVar的對應。
